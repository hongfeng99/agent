# Chapter 4：智能体经典范式构建

本章分别实现了三种经典的 Agent 范式：

1. ReAct
2. Plan-and-Solve
3. Reflection

---

## 1. ReAct

### 核心流程

Reasoning → Action → Observation → Reasoning

ReAct 智能体不会提前生成一份固定计划，而是根据当前任务和工具返回的观察结果，动态决定下一步行动。

### 我的实现

主要组件：

- `HelloAgentsLLM`：调用大模型
- `ToolExecutor`：注册和执行工具
- `ReActAgent`：控制思考、行动和观察循环
- `parse_response()`：解析模型生成的 Thought、Action 和 Action Input

### 执行流程

1. 用户提交问题；
2. 模型判断下一步应该做什么；
3. 模型选择工具；
4. Python 执行工具；
5. 工具结果作为 Observation 返回模型；
6. 模型根据 Observation 决定下一步；
7. 得到 Final Answer 后结束。

### 优点

- 可以动态选择工具；
- 能根据工具结果调整后续行动；
- 适合信息不完整、需要探索的任务；
- 执行过程比较灵活。

### 缺点

- 容易出现重复工具调用；
- 模型输出格式可能不稳定；
- 可能陷入 Thought、Action、Observation 循环；
- 需要设置最大执行步数。

### 适合任务

- 查询天气；
- 搜索景点；
- 调用计算器；
- 文件读取；
- 多工具组合任务。

---

## 2. Plan-and-Solve

### 核心流程

Plan → Execute Step 1 → Execute Step 2 → ... → Final Answer

Plan-and-Solve 将任务明确分成两个阶段：

1. Planner 先生成完整行动计划；
2. Executor 再按照计划依次执行。

### 我的实现

主要组件：

- `Planner`：生成计划列表；
- `parse_plan()`：将模型输出转换成 Python 列表；
- `Executor`：逐步执行计划；
- `AnswerSynthesizer`：综合所有步骤的执行结果；
- `PlanAndSolveAgent`：控制完整流程。

### 执行流程

1. 用户提交问题；
2. Planner 将问题拆分成多个步骤；
3. Executor 执行第一个步骤；
4. 第一个步骤的结果保存到 `previous_results`；
5. Executor 执行第二个步骤，并读取此前结果；
6. 重复执行，直到所有步骤完成；
7. AnswerSynthesizer 综合全部结果；
8. 返回最终答案。

### 优点

- 任务结构清晰；
- 适合多步骤推理；
- 每一步的目标明确；
- 执行过程容易查看和调试。

### 缺点

- 初始计划错误会影响后续步骤；
- 执行阶段默认不会动态修改计划；
- 步骤较多时会产生多次模型调用；
- 简单任务可能被过度拆分。

### 适合任务

- 数学应用题；
- 学习计划；
- 项目实施方案；
- 结构清晰的多步骤分析；
- 可以提前确定步骤的任务。

## 3. Reflection

### 核心流程

Execution → Reflection → Refinement → Reflection

Reflection 智能体先生成初稿，然后评价初稿中的问题，再根据反馈进行修改。

### 我的实现

主要组件：

- `Memory`：保存执行和反思轨迹；
- `execute_task()`：生成初稿；
- `reflect()`：评价当前答案；
- `refine()`：根据反馈修改答案；
- `should_stop()`：判断是否停止迭代；
- `run()`：控制多轮反思循环。

### 执行流程

1. 用户提交任务；
2. 模型生成初稿；
3. 初稿保存为 `execution`；
4. 模型检查初稿并生成反思意见；
5. 反思意见保存为 `reflection`；
6. 判断是否已经“无需改进”；
7. 如果仍需改进，模型生成新的答案；
8. 新答案再次保存为 `execution`；
9. 重复反思，直到满足停止条件或达到最大轮数。

### 优点

- 能够发现初稿中的问题；
- 可以迭代提高答案质量；
- 执行轨迹清晰；
- 适合准确性要求较高的任务。

### 缺点

- 模型调用次数较多；
- 运行时间和成本较高；
- 可能不断提出没有必要的小修改；
- 停止条件依赖模型反馈；
- 自我反思不保证一定发现真正的错误。

### 适合任务

- 代码生成和代码优化；
- 报告修改；
- 文章润色；
- 复杂答案检查；
- 对准确性和完整性要求较高的任务。

## 4. 三种范式对比

| 对比项目 | ReAct | Plan-and-Solve | Reflection |
|---|---|---|---|
| 核心思想 | 边思考边行动 | 先规划再执行 | 执行后反思改进 |
| 是否使用工具 | 通常需要 | 可以不使用 | 可以不使用 |
| 是否提前生成计划 | 否 | 是 | 否 |
| 是否根据中间结果调整 | 是 | 默认较弱 | 通过迭代调整答案 |
| 是否保存历史结果 | 保存消息和 Observation | 保存 previous_results | 保存 execution 和 reflection |
| 主要风险 | 重复调用、无限循环 | 初始计划错误 | 反思过度、成本较高 |
| 典型任务 | 搜索和工具调用 | 多步骤推理和规划 | 代码、报告和高质量输出 |

## 5. 三种范式如何选择

### 选择 ReAct

当任务需要根据外部结果动态决定下一步时，选择 ReAct。

例如：

- 先查天气；
- 根据天气决定推荐室内还是室外景点；
- 再根据用户要求继续查询其他信息。

### 选择 Plan-and-Solve

当任务可以提前拆分为明确步骤时，选择 Plan-and-Solve。

例如：

- 先计算周二销量；
- 再计算周三销量；
- 最后计算总销量。

### 选择 Reflection

当已经能够生成答案，但希望进一步检查和改进质量时，选择 Reflection。

例如：

- 生成 Python 代码；
- 检查代码边界条件；
- 根据问题修改代码；
- 再次检查修改结果。

## 6. 三种范式可以组合使用

实际项目不一定只能选择一种范式。

可以组合成：

用户任务
→ Plan-and-Solve 制定计划
→ ReAct 在执行过程中调用工具
→ Reflection 检查最终结果
→ 输出最终答案

例如旅行规划智能体：

1. Plan-and-Solve 拆分旅行规划步骤；
2. ReAct 调用天气和景点搜索工具；
3. Reflection 检查时间安排是否合理；
4. 最后输出完整旅行计划。

## 8. 本章总结

ReAct 解决的是“下一步应该调用什么工具”。

Plan-and-Solve 解决的是“复杂任务应该按照什么步骤完成”。

Reflection 解决的是“已经生成的答案应该如何检查和改进”。

三种范式并不是互相替代的关系，而是针对不同任务特点采用的不同控制策略。
